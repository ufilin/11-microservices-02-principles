# 11-microservices-02-principles

## Задача 1: API Gateway

  Предложите решение для обеспечения реализации API Gateway. Составьте сравнительную таблицу возможностей различных программных решений. На основе таблицы сделайте выбор решения.

Решение должно соответствовать следующим требованиям:

* маршрутизация запросов к нужному сервису на основе конфигурации,
* возможность проверки аутентификационной информации в запросах,
* обеспечение терминации HTTPS.

### Сравнительная таблица решений API Gateway

| Решение | Маршрутизация | Аутентификация | Терминация HTTPS | Ключевая особенность |
| :--- | :--- | :--- | :--- | :--- |
| **NGINX** | ✅ (конфиг-файлы) | 🔶 (через auth_request) | ✅ | Высокая производительность |
| **Kong Gateway** | ✅ (Admin API) | ✅ (JWT, OAuth, Key-Auth) | ✅ | На базе NGINX + БД |
| **Apache APISIX** | ✅ (динамически, etcd) | ✅ (100+ плагинов) | ✅ | etcd + поддержка Wasm |
| **Traefik** | ✅ (авто-дискавери) | ✅ (Middleware) | ✅ | Интеграция с K8s/Docker |
| **Envoy** | ✅ (xDS API) | ✅ (фильтры) | ✅ | Основа для Service Mesh |
| **Yandex API Gateway** | ✅ (Spec-first) | ✅ (IAM/Lockbox) | ✅ | Серверless, SaaS |
| **AWS API Gateway** | ✅ (VPC Links) | ✅ (Cognito, IAM) | ✅ | Глубокая интеграция с AWS |

> ✅ — полная поддержка требования  
> 🔶 — частичная или требующая доработок

### Выбор решения

**Выбрано:** **Apache APISIX**

**Обоснование выбора:**

1. **Маршрутизация на основе конфигурации**  
   APISIX использует etcd для хранения конфигураций. Изменения маршрутов применяются динамически через Admin API **без перезагрузки**.

2. **Проверка аутентификации**  
   Встроенные плагины: `jwt-auth`, `key-auth`, `basic-auth`, `openid-connect`.

3. **Терминация HTTPS**  
   Полная поддержка TLS/mTLS, управление сертификатами на уровне маршрутов и сервисов.

**Почему не другие решения:**
- **NGINX** — требует reload при изменении конфигурации (простои/риск ошибок).
- **Kong** — тоже хорош, но архитектура на БД медленнее etcd в высокодинамичных сценариях.
- **Traefik** — хорош для Kubernetes, но менее гибок в сложных сценариях аутентификации.
- **Облачные решения (Yandex/AWS)** — привязка к вендору, сложнее переносить между средами.

### Итог

Apache APISIX оптимально закрывает все три требования заказчика в контексте крупной компании, обеспечивая динамическую маршрутизацию, богатую аутентификацию из коробки и терминацию HTTPS без компромиссов по производительности и надежности.

### Обоснование выбора решения

**Выбрано: NATS JetStream**

#### Почему NATS JetStream, а не Kafka или RabbitMQ

##### 1. Кластеризация для обеспечения надёжности

NATS JetStream использует **Raft-консенсус** для репликации метаданных и сообщений. Можно настроить количество реплик на поток (`replicas: 3`), что даёт автоматический failover при отказе ноды [citation:1]. Это полностью соответствует требованию *«поддержка кластеризации для обеспечения надёжности»*.

##### 2. Хранение сообщений на диске в процессе доставки

JetStream — это **персистентный слой** над быстрым NATS. Все сообщения пишутся на диск с поддержкой репликации. При этом сохраняется возможность работать в «эфемерном» режиме для ненадёжных сообщений (Core NATS) — лучший из двух миров [citation:1][citation:5].

##### 3. Высокая скорость работы

NATS JetStream показывает **200 000–400 000 сообщений в секунду** с персистентностью и **субмиллисекундную задержку** (1–5 мс с записью на диск). Для сравнения: RabbitMQ — 50 000–100 000 msg/s, Kafka — 500 000–1 000 000 msg/s, но с задержкой 10–50 мс [citation:1].

**Что это даёт на практике?**  
Kafka быстрее по чистому throughput, но платит за это высокой латентностью. NATS JetStream даёт **лучший баланс**: достаточно высокая скорость + минимальная задержка. Для большинства микросервисных сценариев это важнее, чем абсолютный рекорд по throughput.

##### 4. Поддержка различных форматов сообщений

NATS работает с **сырыми бинарными данными** и не навязывает никаких схем. Это значит, что можно передавать JSON, Protobuf, Avro, MessagePack и любые другие форматы. Серверу всё равно — он просто доставляет байты [citation:7].

##### 5. Разделение прав доступа к различным потокам сообщений

NATS поддерживает **per‑subject permissions** и аккаунты. Можно тонко настроить, кто может публиковать и подписываться на определённые топики (в терминах NATS — subjects) [citation:1]. Для крупной компании с десятками команд и сервисов это критично.

##### 6. Простота эксплуатации — ключевое преимущество

**Это самый сильный пункт NATS.** В отличие от Kafka (которая даже в версии 4.0 без ZooKeeper остаётся тяжёлой JVM‑системой) и RabbitMQ (требующего Erlang/OTP и настройки кластеризации), NATS — это **один бинарник**, который запускается одной командой [citation:6][citation:7].

Для DevOps это означает:
- Минимум зависимостей (не нужны ZooKeeper, BookKeeper, отдельный кластер для метаданных)
- Лёгкое обновление (просто заменить бинарник)
- Понятная конфигурация
- Быстрое восстановление после сбоев

Как пишут разработчики в 2025 году, NATS — это «самая приятная система для использования» [citation:7].

### Почему не подходят другие варианты

| Решение | Почему не выбрано |
| :--- | :--- |
| **Apache Kafka** | Хотя это промышленный стандарт для high‑throughput сценариев, он **сложен в эксплуатации** (даже с Kafka 4.0). Высокая latency и операционная тяжесть делают его избыточным для большинства микросервисных сценариев, не связанных с petabyte‑масштабом [citation:7][citation:9] |
| **RabbitMQ** | Хорош для сложной маршрутизации (exchanges, bindings) и legacy‑систем на AMQP, но имеет **проблемы с кластеризацией** в прошлом [citation:7]. Скорость в 5–10 раз ниже, чем у NATS [citation:1]. Кроме того, после подтверждения сообщения (ACK) оно удаляется — нет возможности replay [citation:9] |
| **Apache Pulsar** | Технологически интересен (storage/compute separation), но **ещё сложнее Kafka**: нужно управлять Brokers + BookKeeper + ZooKeeper [citation:4][citation:5]. Избыточен для требования *«простота эксплуатации»* |

### Итог

NATS JetStream — это **оптимальный баланс** между всеми требованиями:
- ✅ Кластеризация (Raft, реплики)
- ✅ Хранение на диске (JetStream)
- ✅ Высокая скорость + низкая задержка
- ✅ Любые форматы сообщений
- ✅ Разграничение доступа (per‑subject ACL)
- ✅ Простота эксплуатации (один бинарник)

Особенно важен последний пункт: в крупной компании время инженеров — дорогой ресурс. NATS позволяет не тратить его на настройку ZooKeeper и решение проблем с кластером RabbitMQ/Kafka, а сосредоточиться на бизнес‑логике.

**Для какого сценария NATS не подойдёт?**  
Если вам нужна строгая **упорядоченность событий в рамках всей системы** (event sourcing, CQRS с万亿 событий в день), Kafka будет правильнее. Для классической микросервисной архитектуры с десятками сервисов NATS JetStream — идеальный выбор.
