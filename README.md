# 11-microservices-02-principles

## Задача 1: API Gateway

  Предложите решение для обеспечения реализации API Gateway. Составьте сравнительную таблицу возможностей различных программных решений. На основе таблицы сделайте выбор решения.

Решение должно соответствовать следующим требованиям:

* маршрутизация запросов к нужному сервису на основе конфигурации,
* возможность проверки аутентификационной информации в запросах,
* обеспечение терминации HTTPS.

### Сравнительная таблица решений API Gateway

| Решение | Маршрутизация | Аутентификация | Терминация HTTPS | Ключевая особенность |
| :--- | :--- | :--- | :--- | :--- |
| **NGINX** | ✅ (конфиг-файлы) | 🔶 (через auth_request) | ✅ | Высокая производительность |
| **Kong Gateway** | ✅ (Admin API) | ✅ (JWT, OAuth, Key-Auth) | ✅ | На базе NGINX + БД |
| **Apache APISIX** | ✅ (динамически, etcd) | ✅ (100+ плагинов) | ✅ | etcd + поддержка Wasm |
| **Traefik** | ✅ (авто-дискавери) | ✅ (Middleware) | ✅ | Интеграция с K8s/Docker |
| **Envoy** | ✅ (xDS API) | ✅ (фильтры) | ✅ | Основа для Service Mesh |
| **Yandex API Gateway** | ✅ (Spec-first) | ✅ (IAM/Lockbox) | ✅ | Серверless, SaaS |
| **AWS API Gateway** | ✅ (VPC Links) | ✅ (Cognito, IAM) | ✅ | Глубокая интеграция с AWS |

> ✅ — полная поддержка требования  
> 🔶 — частичная или требующая доработок

### Выбор решения

**Выбрано:** **Apache APISIX**

**Обоснование выбора:**

1. **Маршрутизация на основе конфигурации**  
   APISIX использует etcd для хранения конфигураций. Изменения маршрутов применяются динамически через Admin API **без перезагрузки**.

2. **Проверка аутентификации**  
   Встроенные плагины: `jwt-auth`, `key-auth`, `basic-auth`, `openid-connect`.

3. **Терминация HTTPS**  
   Полная поддержка TLS/mTLS, управление сертификатами на уровне маршрутов и сервисов.

### Итог

Apache APISIX оптимально закрывает все три требования заказчика в контексте крупной компании, обеспечивая динамическую маршрутизацию, богатую аутентификацию из коробки и терминацию HTTPS без компромиссов по производительности и надежности.


## Задача 2: Брокер сообщений

Составьте таблицу возможностей различных брокеров сообщений. На основе таблицы сделайте обоснованный выбор решения.

Решение должно соответствовать следующим требованиям:

 - поддержка кластеризации для обеспечения надёжности,
 - хранение сообщений на диске в процессе доставки,
 - высокая скорость работы,
 - поддержка различных форматов сообщений,
 - разделение прав доступа к различным потокам сообщений,
 - простота эксплуатации.

Обоснуйте свой выбор.

### Сравнительная таблица возможностей брокеров сообщений

| Критерий | Apache Kafka | RabbitMQ | NATS JetStream | Apache Pulsar |
| :--- | :--- | :--- | :--- | :--- |
| **Кластеризация / Надёжность** | ✅ ISR-репликация, лидер/фолловер | ✅ Quorum-очереди (Raft) | ✅ R3-кластеризация (Raft) | ✅ BookKeeper, geo-репликация |
| **Хранение на диске** | ✅ Commit Log (сегменты) | ✅ Сообщения, квитанции | ✅ Streams с репликами | ✅ Ledgers (BookKeeper) |
| **Скорость (пропускная способность)** | ★★★★★ (500K‑1M+ msg/s) | ★★★☆☆ (50K‑100K msg/s)  | ★★★★☆ (200K‑400K msg/s)  | ★★★★★ (миллионы msg/s) |
| **Задержка (латентность)** | ★★☆☆☆ (10‑50 мс — из‑за батчинга)  | ★★★★★ (<1 мс до 20 мс)  | ★★★★★ (субмиллисекундная — 1‑5 мс)  | ★★★☆☆ (20‑35 мс)  |
| **Форматы сообщений** | Любые (бинарные) + Avro/Protobuf через Schema Registry | Любые (AMQP, MQTT, STOMP)  | Любые (бинарные) | Любые (бинарные) + Schema Registry |
| **Разделение прав доступа** | ✅ ACL (темы, группы, кластер) | ✅ Vhosts + детальные права | ✅ Subject‑permissions + аккаунты | ✅ Нативная multi‑tenancy  |
| **Простота эксплуатации** | ⚠️ Сложно (Kafka 4.0 без ZooKeeper — проще, но всё ещё тяжело)  | ✅ Просто (веб‑UI, минимум зависимостей) | ✅✅ Очень просто (один бинарник)  | ⚠️ Сложно (Broker + BookKeeper + ZooKeeper)  |


### Обоснование выбора решения

**Выбрано: NATS JetStream**

NATS JetStream — это **оптимальный баланс** между всеми требованиями:
- ✅ Кластеризация (Raft, реплики)
- ✅ Хранение на диске (JetStream)
- ✅ Высокая скорость + низкая задержка
- ✅ Любые форматы сообщений
- ✅ Разграничение доступа (per‑subject ACL)
- ✅ Простота эксплуатации (один бинарник)
